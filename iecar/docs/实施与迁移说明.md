# 实施与迁移说明

## 已落地项

- P0-01：`check_static.sh` 增加 quick/strict 两级验证。quick 覆盖 Python 协议、生成物同步、wrapper 边界、状态回放与 host C 测试；strict 在 quick 后继续执行 `pio run -e car1`、`pio run -e car2`、`pio test -e native`。
- P0-02：car2 Task2 使用 `Task2_RequestCompleteStop` 表示正常完成，`SAFE_STOP` 只保留异常语义。
- P0-03：`tools/protocol_contract.py` 成为单源协议配置；K210/OpenMV wire frame 增加 version byte；CarLink 帧增加 sender role 与 version 校验；文档明确区分 wire frame 和 STM32 RX buffer。
- P1-01：Task2/Task3 进入状态时通过 `task_runtime_state_desc_t` 状态表和 `TaskRuntime_RecordStateFromTable` 统一登记。
- P1-02：OpenMV 首帧闸门、K210 首帧/可靠性门控补齐；策略层拒绝非法 `detect`，避免未定义移位。
- P1-03：CarLink 增加角色矩阵、重复包幂等、越权/坏帧/legacy 计数；兼容旧 task-field 帧时归一化 sender role 后再进入角色矩阵。
- P1-04：新增 host C parser/strategy 测试和 `env:native` 测试入口，覆盖真实共享 parser、CarLink sender role 归一化和策略非法 target 防护。
- P2-01：保留 wrapper 兼容路径，并用 `check_wrapper_boundaries.py` 验证 wrapper 仅含 include、且共享实现 include 必须为最后一行。
- P2-02：文档入口统一为当前 PlatformIO 目录与 v2.3 协议规格。
- P2-03：结构化诊断 ring buffer 增加 `CarDiag_RecordThrottled`，对 no-frame/stale/unreliable/link degraded 等高频故障路径节流。

## 迁移步骤

1. 在开发机安装 PlatformIO CLI。
2. 执行 `./tools/check_static.sh quick`，确认协议生成物、脚本回放、host C 测试全部通过。
3. 执行 `./tools/check_static.sh strict`，确认 `car1`、`car2` 和 native 测试均通过。
4. 若三端协议改动，只改 `tools/protocol_contract.py`，然后运行 `tools/generate_protocol_contract.py` 生成 C/Python 常量，再同步 K210/OpenMV 脚本与 STM32 固件。
5. 现场调参优先改 `car_project_config_shared.h` 或角色 wrapper 配置，不直接修改协议 parser。

## 回滚方式

- 视觉帧临时回滚：启用 `CAR_VISUAL_PROTOCOL_COMPAT_V1` 接收旧无版本 11 字节 K210/OpenMV 帧。
- CarLink 临时回滚：启用 `CAR_PROTOCOL_COMPAT_V1` 和 `CAR_LINK_COMPAT_TASK_FIELD`，旧 task-field 帧会归一化为 expected peer role。
- 任务行为回滚：保留 wrapper 分层，不删除旧角色入口；可按文件级回退 Task2/Task3。

## 验证真实性约束

没有执行过的验证项不得写成已通过。当前脚本会在 strict 模式缺少 PlatformIO 时直接报错，而不是静默跳过。
